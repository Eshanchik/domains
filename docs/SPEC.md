# SPEC — DomainGuard v1.0 (утверждено)

> Статус: **утверждено**, все открытые вопросы закрыты (см. §11 — журнал решений).
> Дата: 2026-07-20. Изменения спеки — только через правку этого файла с записью в §11.

---

## 1. Контекст и цель

Внутренняя система мониторинга доменов для **2 компаний** с несколькими **проектами
(филиалами)** внутри каждой. Текущий объём — **до 1000 доменов**, проектный запас —
**10 000** без переписывания. Домены зарегистрированы у разных регистраторов
(Namecheap, GoDaddy, Hostinger, Marcaria, Regway и др.); аккаунтов у одного
регистратора может быть несколько.

Цели:
- G1. Единая точка правды по доменам с иерархией Компания → Проект → Домен.
- G2. Ни один домен и сертификат не истекает незамеченным.
- G3. Раннее обнаружение проблем репутации (VirusTotal) и падения рабочих URL.
- G4. Автоимпорт где есть API; массовый импорт (список/CSV) где нет.
- G5. Учёт стоимости продлений: сколько платим по компаниям/проектам.

Вне скоупа: автопокупка доменов, DNS-хостинг, email-алерты, PDF-отчёты,
общий uptime-мониторинг «всего подряд» (мониторим только настроенные health-check URL).

Развёртывание: один сервер DigitalOcean (~4 vCPU / 8 GB), Docker Compose.
Бэкапы — снапшоты DO (отдельные pg_dump не требуются; скрипт дампа можно
приложить как опцию, но не автоматизировать).

## 2. Иерархия и модель доступа

**Иерархия: Company → Project → Domain.**
- Домен принадлежит ровно одному проекту; проект — ровно одной компании.
- Теги — дополнительная сквозная группировка (many-to-many).

**Роли (RBAC — сразу, с MVP):**

| Роль | Права |
|---|---|
| Admin | Всё: пользователи, компании, интеграции, секреты, правила алертов |
| Manager | Домены, health-checks, алерты в рамках назначенных компаний/проектов |
| Viewer | Только чтение в рамках назначенных компаний/проектов |

- ACL-1. Скоуп пользователя задаётся списком компаний и/или проектов.
- ACL-2. Все изменяющие действия — в аудит-лог (кто, что, когда, diff).
- AUTH-1. Аутентификация: логин/пароль (argon2), сессии, rate-limit на логин,
  блокировка после N неудач. 2FA — Фаза 2.

## 3. Функциональные требования

### 3.1. Компании и проекты
- FR-CO-1. CRUD компаний: название, код, ответственные, каналы уведомлений по умолчанию.
- FR-CO-2. CRUD проектов внутри компании: название, код, ответственный,
  переопределение каналов уведомлений (опционально).
- FR-CO-3. Теги (сквозные) поверх иерархии.

### 3.2. Домены — реестр
- FR-DM-1. CRUD доменов. Поля:
  - FQDN (+ punycode/IDN автоматически), TLD
  - проект (обязателен) → компания (вычисляется), теги, ответственный
  - регистратор, аккаунт регистратора (если из API)
  - даты: регистрация, **истечение**, обновление
  - auto-renew (да/нет/неизвестно), EPP-статусы, nameservers, registrant (если отдаёт RDAP)
  - источник каждого автозаполняемого поля (manual / csv / api-* / rdap)
  - доп. хосты для SSL-проверки (список, см. FR-CK-2)
  - активен/архив, заметки
- FR-DM-2. Импорт тремя способами: (а) один домен вручную; (б) bulk-вставка
  списком (textarea, по строке на домен); (в) **CSV-импорт** с предпросмотром
  и отчётом (создано/обновлено/ошибки). Формат CSV: `fqdn` обязателен,
  опционально `project_code, tags, notes, renewal_price, currency`.
- FR-DM-3. Дедупликация по FQDN: повторный импорт = upsert, не дубль.
- FR-DM-4. Bulk-операции над выборкой: назначить проект, теги, архивировать; экспорт CSV.
- FR-DM-5. История изменений ключевых полей (expiry, статусы, NS) — таймлайн в карточке.

### 3.3. Учёт стоимости
- FR-CT-1. У домена: цена продления, валюта (по умолчанию USD), период (1 год и т.п.).
- FR-CT-2. **История платежей**: записи «дата, сумма, валюта, курс к USD на дату,
  сумма в USD, комментарий». Записи добавляются вручную или при импорте CSV.
- FR-CT-3. Курс валют: при добавлении платежа не в USD курс подтягивается
  автоматически (открытый API курсов, напр. exchangerate.host/НБУ) с возможностью
  ручной правки; курс фиксируется в записи платежа навсегда.
- FR-CT-4. Сводка расходов: за год/период по компании, проекту, регистратору;
  прогноз ближайших продлений (домены, истекающие в N дней × их цена).

### 3.4. Интеграции с регистраторами
- FR-RG-1. **Аккаунты регистраторов не привязаны к компаниям** — это независимые
  сущности (у одного регистратора несколько аккаунтов; один аккаунт может
  содержать домены разных компаний). Привязка домена к проекту — вручную или
  правилами при импорте.
- FR-RG-2. Плагинный интерфейс `RegistrarConnector` (list_domains → нормализованный
  формат). Порядок реализации: **Namecheap (MVP) → GoDaddy (Фаза 2)** → остальные.
- FR-RG-3. Namecheap: `namecheap.domains.getList` (+ пагинация), expiry, auto-renew.
  Требуется whitelist IP сервера.
- FR-RG-4. Регистраторы без API (Marcaria, Hostinger, Regway): домены заводятся
  импортом; expiry/статусы для них мониторятся через RDAP/WHOIS (регистратор
  указывается вручную или определяется из RDAP).
- FR-RG-5. Синк идемпотентен; поля с источником `manual` не перетираются.
  Новые домены из API попадают в очередь «неразобранные» для назначения проекта.
- FR-RG-6. Учётные данные — шифрованно at-rest (см. §7).
- FR-RG-7. **Цена продления из Namecheap API.** `namecheap.users.getPricing`
  (ProductType=DOMAIN, ActionName=RENEW) отдаёт цену продления за 1 год по TLD.
  Карта TLD→цена кэшируется в Redis (TTL 24ч; меняется редко) и применяется к
  доменам аккаунта (`renewal_price`/`renewal_currency`, источник `api-namecheap`)
  при периодическом синке — не перетирая `manual` (FR-RG-5). Цена берётся по TLD,
  поэтому это оценка (без учёта промо/премиум-доменов). Требует whitelist IP (как
  FR-RG-3). Внешний вызов — через токен-бакет + circuit breaker + retry; ошибка
  помечает данные stale и не роняет воркер.

### 3.5. Проверки (checks)
Каждый тип — отдельный воркер со своим расписанием, rate-лимитом, ретраями.
У домена — `next_check_at` по каждому типу; планировщик ставит в очередь только
созревшие, с джиттером. Результаты — в историю (retention 12 мес.).

- FR-CK-1. **Expiry**: RDAP (основной, серверы из IANA bootstrap с кэшем) →
  WHOIS (fallback, парсинг по TLD через готовую библиотеку). Раз в сутки на домен.
- FR-CK-2. **SSL/TLS**: apex + `www.` автоматически + доп. хосты из карточки
  домена. Дата истечения, издатель, SAN, ошибки цепочки/handshake. Ежедневно.
- FR-CK-3. **VirusTotal**: `GET /domains/{fqdn}` → malicious/suspicious/harmless,
  reputation, категории. **Free-ключ: бюджет 4 req/мин, 500/день** — воркер
  работает от глобальной очереди с токен-бакетом; при 1000 доменов полный цикл
  ~2–3 дня, т.е. каждый домен проверяется примерно раз в неделю. Алерт при
  `malicious ≥ 1` или резком падении reputation.
- FR-CK-4. **Health-checks (кастомная доступность)** — ключевая фича:
  - У домена 0..N health-check'ов. Домены без них по доступности **не проверяются**
    (многие домены намеренно ничего не отдают по корню — это норма, не алерт).
  - Поля health-check'а: URL (любой путь и query, напр.
    `https://domain.com/click?pid=1&offer_id=625`), метод (GET/HEAD),
    follow_redirects (да/нет), ожидаемые статусы (список/диапазон, напр.
    `301,302` или `200-299`), опционально паттерн для заголовка `Location`
    (подстрока или regex — «редирект должен вести на …»), опционально подстрока
    в теле ответа, таймаут (default 10s), интервал (default 15 мин),
    порог алерта — N подряд неудач (default 3).
  - Успех = статус в ожидаемых И (если задан) Location соответствует И (если
    задана) подстрока найдена. Иначе — failure; после N подряд — алерт
    «check down»; при восстановлении — алерт «check recovered», событие гасится.
  - В карточке домена — статус и мини-история каждого health-check'а.
  - Массовое добавление: применить шаблон health-check'а к выборке доменов
    (с подстановкой `{fqdn}` в URL).
- FR-CK-5. **DNS/NS-изменения** — Фаза 2 (фиксация смены NS, алерт — маркер угона).
- FR-CK-6. Устойчивость: ретраи с экспоненциальным backoff, circuit breaker
  на внешний сервис, пометка `stale` вместо потери данных; сбой внешки не
  роняет систему и не рождает ложных алертов.

### 3.6. Алерты и уведомления
- FR-AL-1. Каналы (плагинный интерфейс): **MVP — Telegram**; Фаза 2 — Slack,
  Discord, generic Webhook. **Email не делаем.**
- FR-AL-2. **Маршрутизация**: канал = сущность (для Telegram: chat_id группы;
  бот один, глобальный, токен в настройках). Канал привязывается к компании,
  проекту или помечается глобальным дефолтом. Разрешение при алерте:
  **домен → проект → компания → глобальный дефолт** (первый найденный уровень;
  на уровне может быть несколько каналов — шлём во все). Это позволяет начать
  с одной общей группы и позже развести по компаниям/проектам без кода.
- FR-AL-3. Правила: `expiry ≤ N дней` (пороги 60/30/14/7/1), `ssl ≤ N дней`
  (30/14/7/3/1), `vt_malicious ≥ 1`, `health-check down/recovered`,
  (Фаза 2: `NS изменился`). Пороги/severity переопределяются на уровнях
  global/company/project/domain.
- FR-AL-4. Дедупликация: dedupe-key (домен+правило+порог), состояния
  active/resolved; повторные прогоны не спамят. Переход через следующий порог
  (30 → 14 дней) — новое уведомление.
- FR-AL-5. **Daily digest (MVP)**: раз в день (время настраивается, TZ
  Europe/Kyiv) сводка по каналу: истекающие домены/SSL, активные VT-детекты,
  упавшие health-check'и. Отдельные мгновенные алерты — только для severity
  high (VT-детект, health-check down, expiry ≤ 7); остальное — в дайджест.
  Режимы настраиваются per-канал.
- FR-AL-6. Эскалация — **не делаем** (решение).
- FR-AL-7. Тестовая отправка в канал из UI; журнал отправок (что/куда/когда/статус).
- FR-AL-8. Язык всех уведомлений — русский.

### 3.7. Дашборд (все тексты UI — русские)
- FR-UI-1. Обзор: всего доменов; истекает в 7/30/90 дней; проблемы SSL;
  VT-детекты; упавшие health-check'и; расходы за год — с разбивкой по
  компаниям и проектам.
- FR-UI-2. Таблица доменов: фильтры (компания, проект, тег, регистратор,
  «истекает до», «есть VT-детект», «health down», источник), поиск, сортировка,
  пагинация; экспорт выборки в CSV.
- FR-UI-3. Карточка домена: поля + вкладки: проверки (история), health-checks,
  платежи, изменения, алерты.
- FR-UI-4. Разделы: компании/проекты, аккаунты регистраторов, каналы, правила
  алертов, пользователи (admin), аудит-лог (admin), импорт.

### 3.8. REST API
- FR-API-1. `/api/v1/*` на все сущности; авторизация — сессия (UI) и
  персональные токены (Фаза 2 — полноценные API-токены и внешние вебхуки).
- FR-API-2. UI работает поверх тех же сервисов (логика не дублируется).

## 5. Архитектура

```
Браузер/API ─TLS→ nginx → FastAPI (stateless, xN) ⇄ PostgreSQL
                                   │ enqueue
                                   ▼
                    Redis (broker + cache + rate-limit + locks)
                                   ▲
        Scheduler (по next_check_at, с джиттером) ставит задачи
                                   ▼
   Воркеры: expiry / ssl / vt / healthcheck / registrar-sync / alerter / digest
   Внешние: RDAP/WHOIS · VirusTotal · Namecheap/GoDaddy · Telegram · API курсов
```

- Все проверки и рассылки — через очередь; масштабирование добавлением воркеров.
- Централизованные токен-бакеты в Redis на каждый внешний сервис
  (WHOIS — per-TLD, VT — 4/мин и 500/день, Telegram — лимиты Bot API).
- Docker Compose; при росте — вынос воркеров/БД без изменения кода.

## 6. Нефункциональные требования

- NFR-1. Ёмкость: до 10 000 доменов без переписывания; текущие ~1000 — с запасом.
- NFR-2. Дашборд/таблица < 1 с на выборках до 10k строк.
- NFR-3. Сбой внешнего сервиса не роняет систему (ретраи/backoff/circuit
  breaker/`stale`), перезапуск сервера не теряет данные и не дублирует алерты.
- NFR-4. Идемпотентность всех задач и импорта.
- NFR-5. Структурные логи (JSON), health-эндпоинты (`/healthz`, `/readyz`),
  базовые метрики (глубина очереди, задержки проверок, ошибки внешних API) —
  эндпоинт в формате Prometheus; полноценный стек Grafana — Фаза 2.
- NFR-6. Бэкапы — снапшоты DigitalOcean (решение); опциональный скрипт pg_dump
  в репозитории без автоматизации.
- NFR-7. Retention истории проверок — 12 месяцев.

## 7. Безопасность

- SEC-1. Секреты (ключи регистраторов, VT, токен бота, конфиги каналов) —
  шифрование at-rest (Fernet), мастер-ключ из env, не в БД и не в git.
- SEC-2. Секреты никогда не попадают в логи и ответы API (маскирование).
- SEC-3. TLS на весь внешний трафик (nginx + Let's Encrypt).
- SEC-4. RBAC + скоупы (§2), аудит-лог изменений.
- SEC-5. Rate-limit и защита от брутфорса на логин; argon2 для паролей.
- SEC-6. Whitelist IP сервера в API Namecheap (и др. где требуется).
- SEC-7. Наименьшие привилегии для пользователя БД и контейнеров (non-root).

## 8. Внешние сервисы и лимиты

| Сервис | Назначение | Лимиты / нюансы |
|---|---|---|
| RDAP | expiry, статусы, NS | Основной источник; bootstrap из IANA, кэш |
| WHOIS | fallback | Жёсткие rate-лимиты per-TLD → токен-бакет, риск бана |
| VirusTotal | репутация | **Free: 4/мин, 500/день** → глобальная очередь, цикл ~неделя |
| Namecheap API | автоимпорт | Требует 20+ доменов или баланс $50 + whitelist IP |
| GoDaddy API | автоимпорт (Фаза 2) | key/secret, свои лимиты |
| Marcaria/Hostinger/Regway | — | Без API: импорт списком, мониторинг по RDAP |
| Telegram Bot API | алерты | Лимиты отправки → очередь + ретраи |
| API курсов валют | конвертация в USD | exchangerate.host или НБУ; кэш на сутки |

## 10. Критерии приёмки MVP

- AC-1. Импорт CSV на сотни доменов: корректный upsert, привязка к проектам, отчёт об ошибках.
- AC-2. Автоимпорт Namecheap из ≥2 аккаунтов; ручные правки не перетираются синком.
- AC-3. Для каждого домена автоматически получены expiry (RDAP/WHOIS) и SSL-даты;
  проверки идут по расписанию с джиттером.
- AC-4. VT-воркер укладывается в бюджет free-ключа (подтверждено логами лимитера);
  детект по тестовому «плохому» домену рождает алерт в нужный канал.
- AC-5. Health-check с URL вида `/click?pid=...` и ожидаемым редиректом:
  при подмене ответа (мок) — алерт «down» после 3 неудач, затем «recovered».
- AC-6. Порог expiry 30 дней → алерт в канал компании; повторные прогоны не спамят;
  переход порога 14 дней → новое уведомление. Daily digest приходит по расписанию.
- AC-7. Тесты симулируют недоступность RDAP/VT (таймаут/429/503): данные
  помечаются stale, система живёт, ложных алертов нет. Перезапуск compose
  не приводит к потере данных и дублям алертов.
- AC-8. Сводка расходов по компании/проекту за год совпадает с суммой платежей (в USD).

## 11. Журнал решений (закрытые вопросы)

1. Термин — «Компания»; введён уровень «Проект» (филиал). Иерархия Company→Project→Domain.
2. RBAC и мультипользователи — сразу в MVP.
3. Учёт стоимости — да: цена продления + история платежей + фиксация курса к USD.
4. RegistrarAccount — независим от компаний (общие аккаунты, привязка на уровне домена).
5. DNS/NS-проверки — Фаза 2. Общий uptime — не делаем; вместо него кастомные
   health-check URL с ожидаемым статусом/редиректом (только для настроенных доменов).
6. VirusTotal — только free-ключ; цикл ~раз в неделю на домен.
7. Эскалация — не делаем. Daily digest — в MVP.
8. PDF-отчёты — не делаем (достаточно CSV-экспорта).
9. Бэкапы — только снапшоты DO.
10. Стек: Python/FastAPI/Postgres/Redis/Dramatiq; фронт — HTMX+Jinja2,
    UI позже заменяется через Claude Design (шаблоны держим чистыми).
11. Коннекторы: Namecheap (MVP) → GoDaddy (Фаза 2); Hostinger/Marcaria/Regway — импорт+RDAP.
12. Ёмкость: ориентир 10k доменов (текущие ~1000, 2 компании, ~5+ проектов).
13. Каналы: только мессенджеры (Telegram в MVP; Slack/Discord/Webhook — Фаза 2). Email — нет.
14. Язык UI и алертов — русский. TZ — Europe/Kyiv (в БД UTC). Валюта учёта — USD.
15. SSL: apex + www + ручной список доп. хостов у домена.
16. Retention истории — 12 месяцев.
17. Auth — логин/пароль; 2FA — Фаза 2. Git — GitHub, CI — GitHub Actions.
